## Objective
Extend Live Claude Sharing — originally a Chrome extension for sharing claude.ai conversations — to support Pi and Claude Code sessions via JSONL file watching, with multi-session management, Cloudflare tunnel for public sharing, and a share link in the viewer header.

## Achievements
- **pi-source.js**: New JSONL watcher that discovers Pi (`~/.pi/agent/sessions/`) and Claude Code (`~/.claude/projects/`) sessions, maps thinking blocks/tool calls/tool results to Live Share protocol, streams in real-time via `fs.watch` + tail
- **ws-server.js rewrite**: Multi-session architecture with `Map<sessionId, session>`, owner/viewer roles, visibility controls, built-in Cloudflare Tunnel (`--tunnel` flag), `server_info` message with share URL, debounced `session_list` broadcasts
- **viewer/index.html rewrite**: Session selector sidebar, collapsible thinking/tool_call/tool_result rendering, toggle buttons (💭🔧📋), share link with copy button showing Cloudflare URL
- **server.js updated**: `--tunnel` and `--pi` flags
- **test-server.js rewrite**: 52 assertions across 10 tests, all passing
- **Cloudflare share link**: Header displays `https://xxx.trycloudflare.com` URL with 📋 copy button — server detects LAN IP as fallback, upgrades to tunnel URL when ready
- **Session list spam fix**: Debounced broadcasts for count updates (2s), immediate for structural changes
- **Windows EXE built**: `dist/Live Claude Sharing Setup 1.0.0.exe` (85MB)

## Decisions
1. **Separate pi-source.js vs embedded in server**
   - **Choix**: Standalone script that runs alongside the server
   - **Rationale**: Decoupled — server doesn't need to know about JSONL format, can be started/stopped independently
   - **Alternative rejetée**: `--pi` flag spawns child process, but path issues with OneDrive spaces on Windows

2. **Owner via URL param**
   - **Choix**: `?owner=true` grants full control (see all sessions, toggle visibility)
   - **Rationale**: Simple, no auth system needed for local use
   - **Alternative rejetée**: Separate password-based auth — overkill for this use case

3. **Share link source**
   - **Choix**: Server sends `server_info` WS message with Cloudflare tunnel URL
   - **Rationale**: Server knows the tunnel URL, viewer doesn't need to guess
   - **Alternative rejetée**: LAN IP detection (user explicitly wanted Cloudflare URL, not localhost)

4. **Session list debounce**
   - **Choix**: 2s debounce for message count updates, immediate for register/visibility/disconnect
   - **Rationale**: Pi sessions produce many events → flooding viewers with session_list every 150ms

## Problems
1. **OneDrive path spaces in spawn**
   - **Symptôme**: `--pi` flag child process fails with path containing spaces
   - **Cause**: Git Bash + OneDrive French paths + spawn
   - **Solution**: Use separate `.bat` files in `C:\temp\` as launchers
   - **Prévention**: Keep pi-source as separate launch

2. **Session list spam**
   - **Symptôme**: Hundreds of `session_list` messages per second to all viewers
   - **Cause**: `_broadcastSessionList()` called on every `message_start` and `full_sync`
   - **Solution**: `_broadcastSessionListDebounced()` with 2s window
   - **Prévention**: Always debounce high-frequency broadcasts

3. **Share link showing localhost**
   - **Symptôme**: User saw `http://localhost:3333/` in header
   - **Cause**: Initial implementation used `location.host` client-side
   - **Solution**: Server detects LAN IP → starts tunnel → broadcasts `server_info` with tunnel URL
